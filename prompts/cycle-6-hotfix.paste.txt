C1. Update components/quotes/invite-party-modal.tsx (or whatever the existing modal file is — find it first, don't rename it):

  - Tab 2 label: "By Email" — renders a new form:

  - On submit of Tab 2: call the new server action invitePartyByEmail (below). On success: close modal, refresh page, success toast "Invited <email> as <role>." On error: inline error message in the form (don't toast errors that point to a specific field).

C2. Add a new server action: lib/actions/parties.ts → invitePartyByEmail({ deal_room_id, email, role, full_name })
  Implementation sketch:
    - Validate: caller is broker for this deal_room (existing pattern); email is valid; role in ['mga','insurer'].
    - Use SUPABASE service-role admin client (createClient with service_role key) to:
        Step 1: Look up profile by email — `from('profiles').select('id, role').eq('email', email).maybeSingle()`
        Step 2A (existing user): if profile found, use that profile.id.
        Step 2B (new user): admin.createUser({ email, email_confirm: true, password: <random 16-char>, user_metadata: { role, full_name: full_name || email.split('@')[0] } }).
            The existing handle_new_user trigger creates the profile row. After admin.createUser returns, fetch the new profile.id by email (or use the returned auth user id; profiles.id should equal auth.users.id per the trigger).
            If for any reason the trigger didn't fire (race), do an explicit insert into profiles with the auth user id, email, role, full_name.
    - Insert parties row: { deal_room_id, party_user_id: <profile.id>, role, invited_at: now() }.
        Use upsert-or-ignore semantics (avoid duplicate party rows for the same email+room). If a unique constraint fires, return a friendly "Already invited" error.
    - Insert activities row: event_type 'invited_by_email', event_data { email, role, full_name }.
    - Update deal_room status from 'draft' to 'active' if not already.
    - Return { success: true, party_id, was_new_user: bool }.

C3. RLS implication: profiles RLS allows users to read their own row + profiles of users sharing a deal room. Adding a new profile then immediately adding them to a party for the broker's deal room means subsequent reads work. The admin client bypasses RLS for the writes in step 2B; that's the right pattern (we use this in earlier cycles for seeding). DO NOT loosen the base RLS policies.

C4. UI nice-to-have (only if time permits): show the new party in the Parties list on the deal room detail page using the email if no profile name is set. The existing list query likely joins profiles.full_name; fall back to profiles.email when full_name is null.

C5. NO migration is required. We're using existing tables.
